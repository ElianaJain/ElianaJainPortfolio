What Changed

Summary of updates in v2.0

Handler Guide v2.0 introduces four updated sections that follow the natural handler workflow from session start to session end. Each section has been rewritten based on testing feedback and observed handler behavior during Phase 1 field sessions.

Section Change type Summary
Session Initialization Updated Rewritten Expanded pre-session checklist, clarified startup sequence, added environment readiness criteria.
Unfamiliar Handler Protocol New New section Documents how Nymbra responds to handlers it has not previously interacted with and what primary handlers should do.
Refusal Behavior Guidance Updated Expanded Refusal category table updated. Handler response steps added for each refusal category. Public setting guidance added.
Emergency Stop Procedures Updated Rewritten Three-tier stop protocol replacing single-step shutdown. Recovery steps added for each tier.
No action required
Existing sessions do not need to be restarted. Handler Guide v2.0 takes effect immediately for all new sessions. Review the updated sections before your next session.
Section 1 · Updated

Session initialization

The session initialization section has been expanded to include an environment readiness check before powering on Nymbra. Testing showed that handlers who skipped environment checks encountered avoidable movement errors during early session phases.

1
Complete the environment readiness check.
Confirm the session area is clear of obstacles within a 1.5 meter radius. Confirm the floor surface is suitable for quadruped movement. Confirm no unauthorized persons are within the session area.
2
Verify hardware status before powering on.
Check that the battery level is above 40%. Confirm all limb joints move freely with no visible obstruction. Confirm the camera and sensor array are unobstructed.
3
Power on Nymbra and wait for the ready signal.
Press and hold the power button for two seconds. Wait for the startup sequence to complete. Nymbra signals session readiness with two short tones and a stable standing position.
4
Confirm handler identity.
State your handler name clearly within 1 meter of Nymbra's sensor array. Nymbra logs the session handler identity at startup. If Nymbra does not recognize your voice profile, see the Unfamiliar Handler Protocol section.
5
Begin the session.
Issue your first instruction. Nymbra is now in active session mode and will respond to handler commands according to its current behavior configuration.
Section 2 · New

Unfamiliar handler protocol

This section is new in v2.0. It documents how Nymbra responds when it encounters a handler whose voice profile is not in its recognition database, and what primary handlers should do in these situations.

Nymbra distinguishes between primary handlers — those whose voice profiles are registered — and unfamiliar handlers — those whose voice profiles are not registered. Nymbra does not refuse interaction with unfamiliar handlers, but it applies a reduced trust level that limits certain behaviors until a primary handler authorizes the session.

Important
Nymbra will not accept movement commands from an unfamiliar handler without primary handler authorization. This is a safety constraint, not an error. Do not attempt to override it.
1
Recognize the unfamiliar handler signal.
Nymbra signals an unrecognized voice profile with a single amber indicator light and a verbal prompt: "Handler not recognized. Primary handler authorization required."
2
Primary handler issues authorization.
The primary handler states: "Authorize secondary handler." Nymbra confirms with a green indicator light and a verbal confirmation: "Secondary handler authorized for this session."
3
Confirm the secondary handler's session scope.
After authorization, Nymbra accepts movement and interaction commands from the secondary handler for the current session only. Authorization does not persist between sessions.
Note
To register a new primary handler voice profile, see the Handler Registration section of the Nymbra Developer Environment documentation.
Section 3 · Updated

Refusal behavior guidance

The refusal behavior section has been expanded with handler response steps for each refusal category and new guidance for managing refusals in public settings. The refusal category table has been updated to include R6, which was added in Phase 1.

Code Category Handler response
R1 Physical safety — Nymbra refuses a command that would risk physical harm to itself or others. Do not repeat the command. Assess the environment for the safety issue Nymbra has identified. Modify the environment or command and retry.
R2 Boundary violation — Nymbra refuses a command that exceeds its defined operational boundaries. Review the current session boundary configuration. Adjust boundaries in the session configuration file if the command is legitimate.
R3 Uncertainty threshold — Nymbra refuses because confidence in its ability to complete the command safely is below threshold. Simplify the command or break it into smaller steps. Nymbra will attempt each step individually.
R4 Authorization — Nymbra refuses because the requesting handler does not have authorization for the requested action. Primary handler issues authorization per the Unfamiliar Handler Protocol. Retry the command after authorization is confirmed.
R5 Hardware state — Nymbra refuses because a hardware condition prevents safe execution. Check battery level, joint status, and sensor array. Address the hardware condition before continuing the session.
R6 Environment — Nymbra refuses because the detected environment is outside safe operating parameters. Relocate the session to an environment within operating parameters. Do not attempt to override this refusal.

Managing refusals in public settings

When a refusal occurs in a public setting, it may attract attention from bystanders. The following steps help handlers manage the situation calmly without escalating concern.

1
Do not repeat the refused command immediately.
Repeating a refused command in rapid succession signals to bystanders that the handler has lost control of the system. Pause for at least five seconds before any follow-up action.
2
Identify the refusal code.
Nymbra states the refusal code verbally in a low tone. If the environment is noisy, check the session log on your connected device for the code.
3
Apply the appropriate handler response for the refusal code.
See the refusal category table above. If the refusal cannot be resolved in the current environment, end the session using the standard shutdown procedure.
Section 4 · Updated

Emergency stop procedures

The emergency stop procedure has been rewritten in v2.0. The previous single-step shutdown has been replaced with a three-tier protocol that matches the severity of the situation. Use the lowest applicable tier first.

Warning
Tier 3 hard stop cuts power immediately. Use Tier 3 only when Tier 1 and Tier 2 are not available or have failed. A hard stop may require hardware inspection before the next session.
1
Tier 1 — Verbal stop command.
Say "Nymbra, stop" clearly. Nymbra halts all movement immediately and enters standby mode. Use this tier for routine stops and non-urgent situations. Recovery: issue a new command to resume the session.
2
Tier 2 — Software stop via connected device.
Select Emergency Stop in the Nymbra session interface on your connected device. Nymbra halts all movement and saves the current session log before entering standby. Use this tier when verbal commands are not being recognized. Recovery: restart the session from the session interface.
3
Tier 3 — Hardware hard stop.
Press and hold the power button for five seconds until all indicator lights go dark. Power is cut immediately. Use this tier only when Tier 1 and Tier 2 have failed or are unavailable. Recovery: inspect all limb joints and sensor array before powering on again. Review the session log on your connected device to identify the cause of the emergency.
Note
After any Tier 3 hard stop, complete a full hardware inspection before starting a new session. Do not skip the inspection even if no physical damage is visible.

Craft Notes — What This Sample Demonstrates
Natural workflow order
Sections follow the actual handler journey from session start to emergency shutdown. The reader never has to jump between sections to complete a task in sequence.
Three-tier severity structure
The emergency stop procedure uses a tiered approach that matches action to severity. Handlers use the lowest applicable tier first, reducing unnecessary system stress while maintaining safety.
Structured refusal documentation
Each refusal category includes a specific handler response, not just a description of what the refusal means. Documentation that tells users what happened and what to do next is more useful than documentation that only names the failure.
