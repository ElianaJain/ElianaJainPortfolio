Documenting What the Docs Left Out

How gap analysis, endpoint testing, and engineering collaboration produced accurate REST API documentation for SAP Ariba's procurement and sourcing platform.

Author Eliana Jain
Product SAP Ariba REST APIs
Audience Internal Engineers and Developer Portal Users
Domain Procurement and Sourcing
The Problem
Existing API docs were incomplete, leaving engineers without accurate information to integrate successfully
My Role
Reviewed existing documentation, tested endpoints, consulted engineers, drafted and submitted for review
The Outcome
Accurate, complete API reference documentation published to the SAP Developer Portal
01

Context

SAP Ariba's REST APIs support procurement and sourcing workflows for enterprise customers. Developers — both internal engineers and external customers with Developer Portal access — rely on this documentation to build integrations, automate purchasing workflows, and connect Ariba to third-party systems.

When I inherited the existing API documentation, I found it incomplete. Error codes were undocumented or incorrectly described, authentication requirements were vague, and the gap between what the docs said and what the endpoints actually did created real friction for developers trying to integrate.

02

Process

My approach to API documentation follows a consistent four-step process that prioritizes accuracy over speed.

Step 01
Review existing docs
Audit current documentation for gaps, inaccuracies, and missing error codes or parameters
Step 02
Test the endpoints
Submit real requests to verify documented behavior against actual API responses
Step 03
Consult engineering
Bring findings to engineers to confirm root causes and verify corrections before drafting
Step 04
Draft and submit
Write accurate documentation and submit for technical review before publishing
03

Key Finding: The 403 Error

The most significant gap I found was in how the documentation described the 403 error response. This is a finding that illustrates the broader problem with inherited API docs — generic descriptions that don't reflect what the API actually does.

Before
403 described generically or not at all

The existing documentation either omitted the 403 response entirely or described it with a generic "Forbidden" label that gave developers no actionable information. A developer hitting a 403 had no way to know what was causing it or how to resolve it.

Before
"status": 403,
"error": "Forbidden"
// No further description
After
"status": 403,
"error": "Forbidden",
"cause": "OAuth token invalid
or missing",
"resolution": "Verify OAuth
credentials and reauthorize"
Why This Matters
A 403 in this context is not a generic permissions error. It is specifically an OAuth authorization failure. Without that distinction, a developer could spend significant time debugging the wrong thing — checking endpoint access, user roles, or network configuration — when the actual issue is an invalid or expired OAuth token. Accurate error documentation reduces debugging time and support escalations.
Resolution
Documented the OAuth cause and trigger conditions

After testing the endpoint and confirming with engineering, I updated the 403 documentation to specify that the error indicates an OAuth token issue, and documented the conditions that trigger it. This gave developers a precise diagnosis and a clear path to resolution.

The correction required testing, an engineering conversation to confirm root cause, and a drafted update submitted through the standard review process before publishing to the SAP Developer Portal.

04

Impact

🎯
Accurate error documentation
Error codes now reflect actual API behavior, not generic HTTP status descriptions, giving developers actionable information at the point of failure.
🔍
Verified against real behavior
Every documented response was tested against live endpoints and confirmed with engineering, not inferred from existing copy or assumed from HTTP standards.
🌐
Published to SAP Developer Portal
Documentation reached both internal engineers and external customers with Developer Portal access, serving the full audience for SAP Ariba's procurement and sourcing APIs.
🤝
Engineering-validated process
The review and approval process established a collaborative workflow between documentation and engineering that improved accuracy across the API reference.
Key Takeaway
Inherited API documentation is almost always incomplete. The work of a technical writer is not to transcribe what exists but to verify what is true, find what is missing, and document what developers actually need to succeed.
