How-To Guide · Error Message Writing

How to Write an Error Message Using the Three-Question Structure

Error messages are the most read and least written documentation in any product. This guide covers a repeatable three-question structure that produces clear, actionable, non-blaming error messages every time.

Author Eliana Jain
Audience Technical Writers · UX Writers · Product Teams
Reading time 5 minutes
01 · Why error messages matter

The most read documentation nobody writes carefully

Error messages are documentation. They appear at the exact moment a user needs help, which makes them more important than most content a technical writer produces. Yet they are often written by engineers under deadline pressure, reviewed by nobody, and never updated.

A bad error message creates three problems: the user doesn't know what happened, doesn't know why it happened, and doesn't know what to do next. A good error message answers all three questions in one sentence.

02 · The three-question structure

Every error message must answer three questions

Before writing an error message, answer these three questions. The answers become the message.

Question 01
What happened?
Name the failure specifically. Never write "something went wrong." That tells the user nothing.
Question 02
Why did it happen?
State the cause if known. If unknown, say so honestly. Don't imply the user caused an error they didn't cause.
Question 03
What should they do next?
Give a specific action. If the issue can't be resolved by the user, direct them to support. Never leave them without a next step.
03 · How to write it

Step by step

1
Identify what actually failed
Talk to engineering. Test the scenario yourself. Find out exactly what the system couldn't do — not just the HTTP status code, but the specific action that failed. "File upload failed" is better than "Error 500." "Your session expired" is better than "Unauthorized."
2
Locate the failure accurately
Decide whether the failure was caused by the user, the system, or an external factor. This determines the tone. If the system failed, say so. Don't write copy that implies the user did something wrong when they didn't. Mislocating blame is the most common error message mistake.
3
Write the three answers in order
What happened. Why it happened. What to do next. Not all three need separate sentences — a well-constructed sentence can answer two at once. But all three questions must be answered. If you can't answer question two, say "We're not sure what caused this" rather than omitting it entirely.
4
Apply the rules
Active voice. Present tense. Under 30 words. No exclamation points. No "please." No "something went wrong." No blame. Specific action for the next step. Check every word against this list before publishing.
5
Test it with a real user
Read the error message out loud. If it sounds like a legal disclaimer or a robot, rewrite it. If a user reads it and still doesn't know what to do, rewrite it. The test is simple: can someone act on this message within five seconds of reading it?
04 · The template

A reusable starting point

Use this template as a starting point. Not every message will follow it exactly — but every message should answer the same three questions.

Template
[What failed] could not [action]. [Reason if known]. [Next step].
Incorrect
Something went wrong. Please try again.
What failed? Why? What does "try again" mean? None of the three questions are answered.
Correct
Your file could not be uploaded. Files must be under 10 MB. Compress your file and try again.
What failed: file upload. Why: size limit. What next: compress and retry.
05 · The rules

Non-negotiables for every error message

No exclamation points
A user in an error state is already frustrated. Exclamation points add inappropriate energy to a failure moment.
No "something went wrong"
This phrase communicates nothing. Name the failure specifically every time, no exceptions.
Active voice only
"The server couldn't process your request" not "Your request could not be processed." The subject acts.
Under 30 words
Error messages are read under stress. Short messages are processed faster and acted on more reliably.
No blame without cause
Only imply user error when the user actually caused the error. Misplaced blame erodes trust immediately.
Always give a next step
Even if the error can't be resolved by the user, direct them somewhere. "Contact support" is better than nothing.
06 · Worked examples

The three-question structure applied

Scenario 01 · File upload failure System error

The user tries to upload a file. The upload fails because the file exceeds the size limit.

What happened?
The file could not be uploaded.
Why?
The file exceeds the 10 MB size limit.
What next?
Compress the file and try again.
Final message
Your file could not be uploaded. Files must be under 10 MB. Compress your file and try again.
Scenario 02 · Session expiry Auth error

The user's session expired while they were editing a form. Changes were not saved.

What happened?
Changes could not be saved.
Why?
The session expired while editing.
What next?
Sign back in to continue. Draft is safe.
Final message
Couldn't save changes. Your session expired while you were editing. Your draft is safe. Sign back in to continue.
Scenario 03 · Feature unavailable Service error

A feature is temporarily down. The user tries to use it and gets an error.

What happened?
File upload is temporarily unavailable.
Why?
We are working on it.
What next?
Core features still work. Check back soon.
Final message
File upload is temporarily unavailable. You can still add and update tasks while we work on it. We'll restore file uploads as soon as possible.
07 · Before you publish

Error message checklist

Run every error message through this checklist before it ships.

  • Does the message name the failure specifically?
  • Does it explain why the failure happened, or acknowledge if unknown?
  • Does it give the user a clear next step?
  • Is the voice active?
  • Is the tense present?
  • Is it under 30 words?
  • Does it avoid "something went wrong"?
  • Does it avoid exclamation points?
  • Does it accurately locate the failure — system or user?
  • Can a user act on it within five seconds of reading it?
