Foundation

Core principles

01
Accuracy over completeness

If you are uncertain, say so. Incomplete but accurate documentation is better than complete but inaccurate documentation.

02
Respect the reader's time

Developers read to accomplish something. Get to the point. Every sentence should earn its place.

03
Show, don't describe

Use examples, code samples, and concrete scenarios. Avoid abstract explanations where a concrete one is possible.

04
Distinguish evidence from interpretation

State what is true. Flag what is inferred. Never present an assumption as a fact.

    By content type

    Tone, structure, and rules

    Tone: Conversational, encouraging, and precise. Write as if you are sitting next to the reader walking them through the task. Be warm but efficient. Do not be condescending.

    Number all steps. Never use bullet points for procedural steps. Start each step with an imperative verb. State what the reader should see after each significant step.

    Acknowledge common failure points. If step 3 often fails due to a known issue, say so before the reader encounters it. Do not skip steps because they seem obvious.

    Example step
    ✓  "3. Enter your API key in the Key field and select Save.
    A confirmation message appears at the top of the screen."

✗  "3. Save your API key."
    Required structure
    1
    Overview — What the reader will accomplish and why it matters. Two to three sentences maximum.
    2
    Prerequisites — What the reader needs before starting. List versions, credentials, and dependencies.
    3
    Steps — Numbered steps. One action per step. No compound steps.
    4
    Verification — How the reader knows they succeeded.
    5
    Next steps — Where to go from here. Two to three options maximum.
    Tone register
    Conversational, encouraging, and precise.

    Tone: Neutral, precise, and complete. API reference is not a conversation. It is a contract. Write with zero ambiguity. Avoid personality. Every claim must be verifiable.

    Document every parameter. Do not skip parameters because they seem self-explanatory. Document every error code with its specific cause, not just its HTTP meaning.

    Never say "see above" or "see below." Link to the specific section. If a field is deprecated, say so explicitly and link to the replacement.

    CodeCauseResolution
    403The OAuth token has expired or is invalid.Generate a new token and retry the request.
    429The request exceeds the rate limit for your plan.Check the Retry-After header, then retry.
    500An unexpected server error occurred.Retry the request. If it persists, contact support with the request ID.
    Required per endpoint
    1
    Endpoint — HTTP method and path.
    2
    Description — What it does. One to two sentences.
    3
    Authentication — Credentials required and how to pass them.
    4
    Request parameters — Name, type, required/optional, description, example.
    5
    Response — Schema with field descriptions and success example.
    6
    Error codes — Every possible code with cause and resolution.
    7
    Code sample — Complete, runnable example in at least one language.
    Tone register
    Neutral, precise, and complete. Zero ambiguity.

    Tone: Direct, helpful, and non-blaming. An error message is received at a moment of failure. The reader is already frustrated. Do not make it worse. Do not blame the user. Do not use exclamation points.

    Never use "something went wrong." Name the failure specifically. Locate the failure accurately: if the system failed, say so. Do not imply the user caused an error they did not cause.

    Keep error messages under 30 words. Use consistent vocabulary across all error messages for the same product.

    Template
    [What failed] could not [action]. [Reason if known]. [Next step].

Example:
"Your file could not be uploaded. Files must be under 10 MB.
 Compress your file and try again."
    Three required questions
    1
    What happened? Name the failure specifically.
    2
    Why did it happen? If known, state the cause.
    3
    What to do next? State the resolution. Do not make the reader search for the answer. If the issue cannot be resolved, direct the user to support.
    Tone register
    Direct, helpful, and non-blaming. Under 30 words.

    Tone: Conversational, opinionated, and honest. A blog post is not documentation. It has a point of view. Write with your own voice. Developer audiences respond to honesty and specificity over polish and generality.

    Lead with the specific, not the general. Show your work. Acknowledge limitations and uncertainty. "I am not sure" and "this may not apply to your situation" build more trust than false confidence.

    Do not pad. A 600-word post that says one thing clearly is better than a 1,500-word post that says three things vaguely.

    Lead example
    ✓  "I spent three months documenting an AI assistant and learned
    that the hardest part is not writing about what it does.
    It is writing about what it will not do."

✗  "AI is changing how we think about documentation."
    Recommended arc
    1
    Hook — A specific problem, observation, or question. Not a broad statement about the industry.
    2
    Context — Why this matters and to whom.
    3
    Body — The argument, walkthrough, or narrative. Use subheadings every 300 to 400 words.
    4
    Conclusion — What you learned or recommend. Be direct.
    Tone register
    Conversational, opinionated, and honest.

    Tone: Neutral and precise. Release notes are a record, not marketing copy. Do not editorialize. State what changed and why it matters to the reader.

    Be specific about what changed. Note breaking changes explicitly and prominently using a Breaking Change label. Keep each entry to 120 to 150 words maximum.

    Active voice throughout: the product performs the action.

    Template
    **[Feature or fix name]**
[One sentence: what changed.]
[One sentence: who is affected and under what conditions.]
[One sentence: action required, or "No action required."]

✓  "Added support for webhook event filtering."
✗  "We are thrilled to announce an exciting new feature!"
    Required per entry
    1
    What changed — The specific feature, behavior, or fix.
    2
    Who it affects — Which users, plans, or configurations are impacted.
    3
    What to do — Any required action. If none, write "No action required."
    Tone register
    Neutral and precise. 120 to 150 words per entry.

    For AI agents

    AI agent instructions

    This style guide is designed to be used as a system reference by AI agents. The following directives apply when generating developer-facing content using this guide.

    directives — apply when generating content
    01
    Apply the correct content type rules based on what is being written. Do not apply API reference rules to a tutorial.
    02
    Check every sentence against the banned words list before finalizing output.
    03
    Check every sentence for em dashes and remove them.
    04
    Verify all error messages follow the three-question structure: what happened, why, what to do next.
    05
    Verify all code samples include language identifiers and explanatory comments.
    06
    Flag any claim that cannot be verified as uncertain rather than stating it as fact.
    07
    Keep all sentences under 25 words. Flag any sentence that exceeds this limit.
    08
    Do not add information not present in the source material. If information is missing, say so explicitly.
    09
    Use active voice. Rewrite passive constructions in active voice before finalizing output.
    10
    When uncertain about tone, default to precise and neutral rather than conversational.
